# How Chronos Projections and vLLM Work in This Setup

This system has two model-serving paths with deliberately separate responsibilities:

1. Chronos-2 generates the numeric time-series projection.
2. vLLM generates language around that projection: explanations, recommendations, and optional note-derived context.

vLLM does not forecast the time series. It never serves Chronos, retrains Chronos, adjusts numeric forecast values, or post-processes quantiles. The authoritative numeric output comes from `ml_service/`; the language output comes later from `llm_service/` through the orchestrator.

## Chronos projection path

The public API request lands at `orchestrator_api/`. For `POST /predict`, the orchestrator validates the request and forwards the model-relevant fields to the Chronos ML service at `http://127.0.0.1:8081/predict`. For `POST /predict/csv`, it first parses the uploaded CSV into the same contract.

The target forecast contract is one series at a time:

- historical target values, required;
- optional historical timestamps;
- forecast horizon, `prediction_length`;
- optional historical covariates, `past_covariates`;
- optional known-future covariates, `future_covariates`;
- optional future timestamps used as output labels;
- requested quantile levels, defaulting to `[0.1, 0.5, 0.9]`.

The configured production model is `autogluon/chronos-2-small`, pinned to revision `ddec01313e50b6bc58ebaa92ede81bc24a3d9f9a` through `chronos-forecasting==2.3.1`. The ML service loads it through `BaseChronosPipeline.from_pretrained(...)` using CPU float32 and the configured Hugging Face cache. `ML_PRELOAD_MODEL=true` causes the service to load the checkpoint at startup before it accepts normal traffic.

Chronos-2 is used zero-shot. We do not train, fine-tune, fit AutoGluon on the incoming dataset, or maintain customer-specific model state. The submitted history and covariates are inference inputs only.

Internally, the Chronos-2 adapter converts the request into the dataframe shape expected by `pipeline.predict_df(...)`:

- `context_df` contains an internal item id, an internal regular daily timestamp, the target values, and any historical covariates.
- `future_df` is included only when known-future covariates are supplied. It contains the future internal timestamps and the future covariate values.
- The model is called with `prediction_length`, sorted quantile levels, and `freq="D"`.

The service intentionally uses an internal regular daily index starting at `2000-01-01`. Supplied timestamps remain the labels returned to the caller, but they are not the index Chronos-2 models against. If supplied timestamps are irregular or unparseable, the service warns that observations are being modeled in their given order as equally spaced points. This keeps the model input regular and predictable while preserving the user's original labels in the response.

Chronos-2 returns forecast rows with quantile columns. The ML service converts those into `ForecastPoint` objects:

- `step`;
- optional response timestamp;
- `median`, always from `q50`;
- `lower` and `upper`, derived from the minimum and maximum requested quantiles;
- full `quantiles`, such as `q10`, `q50`, and `q90`.

The service validates that the horizon length is exact, all quantile values are finite, and quantiles are ordered. If the model fails to load, prediction fails, or the returned shape is invalid, the service falls back to a deterministic trend model and marks `engine="fallback"` and `model_family="fallback"`. That fallback is intentionally simple: it extrapolates recent deltas with damping and widens intervals based on recent volatility. It ignores covariates and includes a warning when it does so.

The retained original Chronos adapter, typically `amazon/chronos-t5-small`, is only a rollback path. It uses `pipeline.predict(...)` to generate samples and then computes quantiles from those samples. It is univariate in this application, so any supplied covariates are ignored with a warning.

After numeric inference, the ML service adds a lightweight operational summary and heuristic drivers. These are not generated by vLLM. They are deterministic fields derived from the observed values and forecast horizon, including trend direction, confidence, risk band, recent volatility, and recent level shift. The complete ML output is the source of truth for downstream language generation and optional database logging.

## vLLM path

vLLM runs as a separate OpenAI-compatible chat-completions endpoint. In the Terraform two-VM setup, it is a private CPU VM exposing port `8000` only to the orchestrator network path. The service is started with:

```text
vllm serve ${VLLM_MODEL} --host 0.0.0.0 --port 8000 --api-key ${VLLM_API_KEY} --dtype bfloat16
```

The orchestrator reaches it through `VLLM_BASE_URL`, for example `http://<vllm-private-ip>:8000/v1`, and sends OpenAI-compatible `POST /chat/completions` requests. The default local configuration names `meta-llama/Llama-3.1-8B-Instruct`; the Terraform example uses `Qwen/Qwen3-0.6B`. In either case, vLLM is just the serving runtime for the selected language model.

The vLLM client performs three language tasks:

1. `extract_structured_features(notes)` turns optional user notes into compact JSON keys such as `signals`, `constraints`, `events`, and `urgency`.
2. `generate_explanation(ml_output, notes)` produces one concise paragraph for an operations stakeholder.
3. `generate_recommendations(ml_output, notes)` produces three practical next actions.

Each prompt includes the compact ML forecast JSON and explicit instructions not to invent data beyond the provided forecast. The client also strips `<think>...</think>` reasoning markup before returning text, because some reasoning-tuned models emit that format.

vLLM output is therefore advisory presentation text. It can explain trend, uncertainty, risk band, and drivers that are already present in the ML output, but it is not allowed to become the numeric forecast authority. If vLLM is unavailable, returns an empty response, times out, or fails authentication, the orchestrator still returns the Chronos forecast. It adds warnings and uses deterministic template text for the explanation and recommendations.

## End-to-end responsibility boundary

The request flow is:

```text
Client
  -> orchestrator_api :8080
     -> ml_service / Chronos-2 :8081
        -> numeric forecast, quantiles, summary, drivers, warnings
     -> vLLM OpenAI-compatible endpoint :8000/v1
        -> explanation, recommendations, optional note features
     -> optional Oracle Autonomous Database logging
  -> combined JSON, Markdown, or enriched CSV response
```

This boundary is important operationally. If Chronos falls back, the numeric forecast should be treated as degraded even if vLLM text is available. If vLLM falls back, the numeric forecast may still be valid; only the generated language companion is degraded. The response fields make this explicit through `ml_output.engine`, `ml_output.model_family`, `loaded_public_model`, `explanation.available`, `recommendations.available`, `used_fallback`, and warnings.
